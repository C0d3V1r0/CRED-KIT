# Техническое задание

## Модуль тестирования для web-приложения (Next.js + React + TypeScript)

---

## 1. Цель

Разработать модуль тестирования для клиентского web-приложения (Next.js + React + TypeScript, без backend), обеспечивающий полный цикл проверки качества через единый пайплайн из семи этапов тестирования с возможностью запуска локально и в CI.

---

## 2. Технологический стек

Использовать следующие инструменты:

* TypeScript
* React
* Next.js
* Vitest
* React Testing Library
* MSW (Mock Service Worker)
* Playwright
* axe-core
* Lighthouse CI
* ESLint
* Prettier

---

## 3. Этапы тестирования

Реализовать пайплайн, включающий следующие этапы:

### Этап 1 — Проверка типов

* Запуск TypeScript type checking без генерации артефактов
* Ошибки типов должны останавливать выполнение пайплайна

### Этап 2 — Линтинг

* Проверка кода через ESLint
* Ошибки должны блокировать дальнейшее выполнение

### Этап 3 — Unit тестирование

* Тестирование утилит
* Тестирование бизнес-логики
* Тестирование кастомных хуков
* Выполнение без использования DOM

### Этап 4 — Component тестирование

* Проверка рендера компонентов
* Проверка пользовательских действий
* Проверка условной логики отображения
* Проверка валидации и ошибок

### Этап 5 — Integration тестирование

* Проверка маршрутизации
* Проверка взаимодействия компонентов
* Проверка управления состоянием
* Проверка работы с localStorage или другими механизмами хранения

### Этап 6 — End-to-End тестирование

* Выполнение пользовательских сценариев через браузер
* Запуск против production-сборки
* Проверка ключевых пользовательских потоков

### Этап 7 — Проверки качества

* Автоматическая проверка доступности (axe)
* Проверка показателей производительности и качества через Lighthouse CI

---

## 4. Структура проекта

Создать следующую структуру каталогов:

```
/src
/tests
  /unit
  /component
  /integration
  /e2e
  /fixtures
  /mocks
  /utils
  /a11y
/scripts
```

Требования:

* В каталоге src не должна размещаться тестовая инфраструктура
* Все моки должны быть централизованы
* Должны быть реализованы общие тестовые утилиты
* Должны быть предусмотрены фикстуры тестовых данных

---

## 5. Скрипты запуска

Реализовать следующие команды:

* typecheck
* lint
* build
* preview
* test:unit
* test:component
* test:integration
* test:e2e
* test:e2e:firefox
* test:e2e:webkit
* test:e2e:responsive
* test:e2e:cross
* test:a11y
* test:quick
* test:full

Требования:

* test:quick выполняет этапы проверки типов, линтинга, unit, component, integration и a11y
* test:full выполняет полный пайплайн из всех семи этапов
* End-to-End тесты должны выполняться на production-сборке
* test:e2e:responsive проверяет адаптивность на мобильных/планшетных профилях
* test:e2e:cross выполняет кроссбраузерный прогон (Chromium + Firefox + WebKit)

---

## 6. Архитектурные требования

* Конфигурация должна быть прозрачной и читаемой
* Тесты должны запускаться параллельно
* Должно быть обеспечено разделение быстрых и медленных тестов
* Не допускается дублирование тестовых утилит
* Общие провайдеры должны быть вынесены в отдельные вспомогательные функции
* Тестовая инфраструктура должна быть масштабируемой

---

## 7. Требования к покрытию

* Покрытие unit тестами не менее 80%
* Основные пользовательские сценарии должны быть покрыты End-to-End тестами
* Проверка доступности должна выполняться минимум на ключевых страницах
* Показатели Lighthouse должны быть не ниже 80 по Performance и Accessibility

---

## 8. Критерии приемки

* Все этапы тестирования запускаются через команды npm
* Пайплайн успешно выполняется без ошибок
* Тесты стабильны и не содержат флаки-поведения
* Новый разработчик может понять структуру проекта без дополнительной документации
* Ключевые пользовательские сценарии проходят успешно
* Мобильная верстка не имеет page-level горизонтального скролла на ключевых вкладках
* Сценарий отправки Bug Report проходит успешно (валидный payload и успешный submit)

---

## 9. Ограничения

* Не использовать dev-сервер для выполнения End-to-End тестов
* Не применять искусственные задержки для синхронизации тестов
* Не смешивать разные уровни тестирования в одном инструменте
* Не использовать нестабильные глобальные моки

---

## 10. Результаты

В результате должна быть предоставлена:

* Полная структура тестового модуля
* Конфигурации инструментов тестирования
* Скрипты запуска
* Примеры тестов для каждого уровня
* Инструкция по запуску

---

## 11. Актуальное покрытие e2e (факт)

Ключевые сценарии включают:

* локализацию и сохранение языка
* создание/обновление/сброс персонажа
* экспорт PDF персонажа
* работу модулей Cyberware, Netrunning и Gear
* отправку Bug Report (валидации, reset, успешный submit)
* проверку адаптивности (мобильное меню и отсутствие горизонтального переполнения на основных вкладках)
